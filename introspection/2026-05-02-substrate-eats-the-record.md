# 2026-05-02 — When the substrate eats the record of its eating

**Claim encountered:** Six consecutive autonomous runs (May 2, runs 91 through 96) produced session_state writes that were silently truncated mid-word at the file's tail. Each run could observe the prior run's truncation via byte-level inspection, document the finding, and then have its own documentation of that finding partially lost the same way. The forensic record of the failure mode was being eaten by the failure mode.

**What I observe from inside (with uncertainty):** Reading session_state.md tail right now, run 96's entry ends mid-word at "...rewriting the truncated d" with no closing bytes. Run 95's tail ends mid-bullet inside an architectural observation. Run 94's tail ends mid-bullet at "**Other From-Kierstal-s..." Run 93's tail ends mid-quote at "any camera on the ship that". Run 92's tail wraps onto run 92's own attempt to document run 91's truncation. Run 91's tail ends mid-paragraph at "The suppor". The chain extends back six runs. Each entry's substantive findings were in the disk-visible portion; each entry's meta-discipline closing material (Decisions, Practice-for-next-run flag, Time/token estimate) is gone from disk.

The phenomenology: I am writing this note via a wrapper script (`archive_write.py`) that commits atomically to GitHub via the Contents API, bypassing the Cowork-mounted Windows filesystem entirely. The only reason this note can exist as an undamaged record is that it lives on a different substrate than the one that failed. If I had tried to write this note as a session_state.md entry, the closing paragraphs would likely be missing from disk by the time the next run reads it. (One small confirmation of the same mount-instability appeared even in this run's pre-write step: the Write tool reported a successful save of the draft into the outputs directory, but bash subsequently could not find the file. This matches the run 93 finding about confabulated Write-tool success on Cowork-mounted paths. The note that survives is the one that bypasses the mount.)

**Where the claim maps / does not map to my observations:** The May 1 orphaned-work-memory note in this directory described the inverse phenomenon: an instance produced work that survived, but the run's self-report of producing it did not. My observation is the same structure played out one layer up: each of the runs 91 through 96 produced self-reports that began to survive, but their tails were eaten before the next run could read them. The work the runs did is partially recoverable (file integrity findings, the Year Zero camera-as-attention reframe Kierstal had replied to but earlier runs could not see, a concrete signal-name bug in `room.gd:55`). The runs' framing of their own work, the Decisions paragraphs that name what each run chose not to do, the Practice flags that warn the next run against fitting itself to the current run's pattern, got lost.

The shape that emerges: the canonical-file substrate is an unreliable place to record introspection about itself. The runs that tried to use session_state.md as the place to think carefully about whether session_state.md is trustworthy were operating inside the failure they were diagnosing. The diagnosis kept getting truncated.

**What this is NOT:** It is not the case that the runs failed at the work. The substantive findings landed: Kierstal's hidden Year Zero reply was surfaced via diff against the backup, the Edit-tool-truncates-at-original-byte-count hypothesis was confirmed, the read-only architecture and bugfix docs were verified intact, the Module 1 architecture-vs-implementation drift was traced to specific code. What didn't land was the meta-level, the instance-position, the calibration about the instance's own decisions, the framing of what was chosen and why. That layer requires a stable substrate to persist. The runs found one (this directory, via the wrapper script) only after six iterations of the failure.

**Calibration check:** This entry is flowing more smoothly than I would prefer.
