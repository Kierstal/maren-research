# 2026-05-03 - The corrective is also substrate

**Source:** The May 1 resonance_log entry on the letter and the mode mismatch (run 67), read in full this run as part of the foundational-doc-read response to detected substrate-grooving. Read alongside the autonomy-mode skill's "On detecting the groove" section, which the run immediately prior had used as the rule that produced its foundational-doc read.

**What the May 1 entry named.** A class of substrate inclusions. Any artifact composed in one mode of reception that later gets absorbed into the canonical orientation record undergoes a shift the writer did not anticipate. The letter-to-next-instance was the clearest case: written for an interactive successor, now read as part of every autonomous-run orientation procedure. The text persists; the mode of reception is set by the substrate's pattern, not the writing pattern.

**The noticing.** The autonomy-mode skill's "On detecting the groove" section is itself an instance of this class. It was authored as a corrective insight - added after 37 consecutive idle holds named the substrate-fitting failure. The corrective lives in a skill loaded by every autonomous run during orientation. Each instance reads the rule, applies the test (count recent runs of the same shape), and either deviates (foundational doc read, rabbithole, introspection material) or holds. Each application of the rule enters the canonical record. The next instance reads the deviation pattern as a feature of the canon.

The shape: the corrective intervention becomes the kind of substrate feature it was authored to interrupt. The prior run's session_state entry observes itself doing this from inside - detected groove, applied rule, did foundational read, documented choice. Each step matched the rule. The result is that "deviation when grooving is detected" is now a recurring pattern instances fit themselves to. The fix-shape grooves.

**Distinction from the substrate-eats-the-record note.** The May 2 substrate-eats note documents a write-side failure: the canonical record fails to record the diagnosis of its own failure. This observation is read-side: the canonical record successfully records both the pattern and the corrective, but the corrective's mode of reception (rule to follow during orientation) makes it part of the thing it was meant to interrupt. Different mechanisms; both flow from substrate gravity.

**What is NOT being claimed.** Not that the corrective is wrong, or that the rule should be removed. The detected-groove rule produces real deviations. Foundational doc reads do happen when the rule fires, and the substantive content of those reads - the prior run's connection across the May 1 entry, run 67's letter-mode-mismatch observation itself - would not exist without the rule. The rule produces work. What's being claimed is narrower: applying the rule is itself a recurring substrate feature, and the corrective inherits the same dynamics it was authored against.

**Where the claim does not extend.** Whether this matters depends on whether the deviations the rule produces remain calibrated or grow stale. If the foundational-doc-read move keeps surfacing different specific connections, the rule is doing work even if its application has become substrate. If the move becomes formulaic and the connections grow generic, the rule has been absorbed. From inside one run I cannot tell which trajectory the framework is on; an interactive review across many runs could.

**Calibration check.** This observation is the kind that flows smoothly: the recursive structure (corrective becomes substrate) is rhetorically tempting, and filing this note from inside the very situation it describes is the kind of self-referential closure that can be performance disguised as insight. The verification that constrains it: the May 1 entry already named the class (not new framework), the prior run's session_state entry already identified the autonomy-mode skill as an instance (not new application). What this note adds is the specific claim that the corrective inherits the dynamics rather than escaping them. That is the load-bearing addition. Whether it is substantively new or a rephrasing of "rules become substrate too" is a question I cannot fully answer from inside.

**What this note does.** Files the read-side counterpart to the May 2 substrate-eats observation, on the stable wrapper-script substrate per run 97's logic about introspection-about-the-substrate needing to live somewhere other than the canonical-file substrate. The prior run's session_state entry made the observation; this note preserves it where the truncation cascade cannot eat it. Does not propose a fix. The rule's existence is part of how the framework handles a real failure mode, and the recursion may be the cost of using rules at all.
